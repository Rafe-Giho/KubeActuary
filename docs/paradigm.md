# KubeActuary Paradigm

## Thesis

AI should not operate Kubernetes through raw command execution. It should operate
through evidence-carrying operation capsules.

The capsule is the unit of intent, review, execution, rollback, and audit.

## New Operating Model

Current model:

```text
human prompt -> AI reasoning -> kubectl/tool calls -> cluster state changes
```

KubeActuary model:

```text
human/AI intent
  -> operation capsule
  -> evidence collection
  -> policy/risk gate
  -> bounded execution
  -> post-change proof
  -> retained audit record
```

## Invention Patterns Used

### Mimic

Borrowed patterns:

- GitOps: declarative artifact before execution.
- Change-management systems: approval, blast radius, rollback.
- Proof-carrying code: execution is allowed only with required proof.
- Kubernetes admission control: validate before mutation.
- SARIF/static-analysis reports: machine-readable findings.

### Synthesis

Combined pieces that usually live separately:

- `kubectl` intent parsing
- `kubectl auth can-i`
- server-side dry-run
- diff
- policy checks
- runtime health checks
- rollback manifest
- AI/MCP tool boundary

The synthesis is not another chat assistant. It is a narrow execution substrate
that any assistant can use.

### Subtraction

KubeActuary deliberately removes:

- built-in LLM dependency,
- automatic write execution in the first version,
- broad natural-language chat UI,
- cluster-admin assumptions,
- hidden mutable session state.

This makes the first primitive auditable and portable.

## Core Object: OperationCapsule

An operation capsule answers:

- What is the requested intent?
- Which command or manifest would be used?
- What Kubernetes target is affected?
- What is the estimated risk?
- What evidence is required before execution?
- What rollback exists?
- What post-checks prove success?
- Who or what created the request?

The capsule can be:

- generated by an AI assistant,
- reviewed by a human,
- checked in to Git,
- verified in CI,
- executed by a future controller,
- retained for audit.

## Gate Semantics

KubeActuary separates three states:

- `drafted`: intent exists, but execution is not justified.
- `evidence-attached`: some evidence exists, but the capsule is still missing
  required proof.
- `verified`: every required evidence item is attached and marked successful.

The execution gate opens only for `verified` capsules. A failed evidence item
sets the capsule state to `blocked`.

## Why This Can Matter

For AI-assisted Kubernetes, the scarce resource is not another interface to
`kubectl`. The scarce resource is trust at the execution boundary.

KubeActuary turns AI from an unconstrained command runner into a producer of
reviewable operational contracts.

## MVP Roadmap

1. Draft capsules from `kubectl` commands and manifests.
2. Verify required evidence is attached.
3. Attach manual or external evidence to capsules.
4. Open or close a local execution gate.
5. Collect real evidence with read-only commands:
   - `kubectl auth can-i`
   - `kubectl diff`
   - `kubectl apply --dry-run=server`
6. Add policy adapters:
   - Kyverno CLI
   - OPA/Rego
   - kube-linter
   - kube-score
7. Add optional MCP server:
   - `draft_operation_capsule`
   - `inspect_operation_capsule`
   - `verify_operation_capsule`
   - later: `execute_approved_capsule`
8. Add admission/controller mode for clusters that want every AI-originated
   change to reference a capsule ID.

## Non-goals

- Replace `kubectl`.
- Replace GitOps.
- Replace policy engines.
- Hide operational risk behind a chat UI.
- Let an LLM self-approve high-risk actions.
